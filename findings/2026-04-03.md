Blog Digest
-----------

**[Harnessing Claude's Intelligence](https://claude.com/blog/harnessing-claudes-intelligence)** (April 2, 2026) — Lance Martin from the Claude Platform Team presents three patterns for building applications that scale with Claude's evolving capabilities: (1) use general-purpose tools Claude already knows well (bash, text editors) rather than domain-specific ones, (2) let Claude orchestrate its own actions, manage its own context, and persist its own memory rather than doing it for him, and (3) set boundaries carefully around caching, security, and observability. Key benchmarks: Opus 4.6 with self-filtering improved BrowseComp from 45.3% to 61.6%; memory folders lifted Sonnet 4.5 accuracy from 60.4% to 67.2% on BrowseComp-Plus.

Suggested Improvements
----------------------

### Skills

**Retire assumptions about context anxiety in skill prompts** — What: Audit skills for workarounds that compensate for model limitations that may no longer exist (e.g., explicit instructions to "not wrap up early," manual context-reset triggers, overly prescriptive step-by-step sequencing that exists to prevent the model from losing track). Why: The post explicitly notes that "context anxiety" behaviors (like Sonnet 4.5 wrapping tasks prematurely) disappeared with Opus 4.5, making compensatory mechanisms "dead weight." How: Review skills — especially long-running ones like `subagent-driven-development`, `executing-plans`, and `software-engineer` — for instructions that exist to work around older model weaknesses rather than to express genuine workflow requirements. Remove or simplify where the instruction was a crutch, not a design choice. Impact: medium. Confidence: likely.

**Review skill frontmatter descriptions for progressive disclosure** — What: Ensure every skill's YAML frontmatter `description` field is specific enough for Claude to decide relevance without loading the full content. Why: The post emphasizes that skills work best as progressive disclosure — brief descriptions in frontmatter, full content loaded on demand. Every token pre-loaded into the system prompt "depletes attention budget." If descriptions are vague ("use for coding tasks"), Claude either loads skills unnecessarily or misses relevant ones. How: Audit each skill's `description` field. A good description names the trigger conditions precisely (e.g., "Invoked when writing Bash scripts or shell automation" vs "use for bash"). Impact: medium. Confidence: likely.

### Claude Code Settings / AgenC Config

**Optimize system prompt structure for prompt caching** — What: Restructure CLAUDE.md content ordering so static, stable content comes first and dynamic/session-specific content comes last. Why: The post details that cached tokens cost 10% of base input and that cache hits require prefix matching. The user's global CLAUDE.md is large and mixes stable conventions (markdown rules, git rules, coding standards) with more volatile content (specific skill references, repo-specific workflows). Reordering so the most stable content leads maximizes cache hit rate across turns. How: Move foundational rules (markdown conventions, git command rules, sandbox rules, coding conventions) to the top of CLAUDE.md. Move skill-specific routing instructions and workflow descriptions lower. Avoid editing the system prompt mid-conversation when possible — use `<system-reminder>` in messages instead. Impact: high. Confidence: likely.

**Evaluate whether the PreToolUse hook for git -C could become a tool-search pattern instead** — What: The `remove-redundant-git-c.sh` hook fires on every PreToolUse event to check for redundant `git -C` usage. Consider whether this could be handled via a declarative tool or prompt instruction instead. Why: The post recommends promoting bash patterns to dedicated tools when you need observability or security boundaries, but keeping bash for general-purpose work. A PreToolUse hook that modifies commands on every invocation adds latency to every tool call. If the instruction in CLAUDE.md is sufficient to prevent the pattern, the hook may be unnecessary overhead. How: Test whether removing the hook and relying solely on the CLAUDE.md instruction produces the same behavior. If Claude still occasionally uses `git -C` for the current directory, keep the hook. If not, retire it. Impact: low. Confidence: speculative.

### Workflows

**Let Claude self-filter tool outputs in research-heavy workflows** — What: When skills or workflows involve processing large tool outputs (e.g., `WebFetch` results, large file reads, git log output), consider giving Claude explicit permission to pipe and filter rather than consuming raw output. Why: The post shows a 16-percentage-point accuracy improvement (45.3% to 61.6%) when Claude could filter its own tool outputs rather than processing everything as tokens. The user's current setup already uses subagents for isolation, but the principle of letting Claude pre-process tool results before consuming them could be applied more broadly. How: In skills that involve large data processing (e.g., this blog processor, research-heavy workflows), add guidance like "use bash to filter and transform large outputs before analysis rather than reading raw results." Impact: medium. Confidence: likely.

**Audit subagent usage patterns against the compaction vs. subagent tradeoff** — What: Review when the setup uses subagents (fresh context windows) vs. compaction (summarizing within the same context). Why: The post presents data showing subagents improved BrowseComp by 2.8%, and compaction with Opus 4.6 reached 84% vs Sonnet 4.5's 43%. The user's setup uses subagents heavily (request review subagents, dispatching parallel agents, subagent-driven development) but may not be optimally choosing between "spawn a subagent" and "let Opus compact and continue." With Opus 4.6's strong compaction, some subagent patterns may be unnecessary overhead. How: For tasks where subagents are spawned primarily to avoid context pollution (not for parallelism), test whether Opus 4.6 can handle the same work in the main context with compaction. Impact: medium. Confidence: speculative.

### Speculative Ideas

**Memory folder pattern for long-running AgenC missions** — What: The post describes Claude writing context to files for later retrieval as a "memory folder" pattern that improved accuracy significantly. The user's current setup explicitly disables auto-memory. However, for long-running missions specifically (not cross-session memory), a mission-scoped scratch directory where Claude can offload intermediate findings could improve multi-step task quality. Why: The 60.4% to 67.2% accuracy improvement on BrowseComp-Plus suggests file-based working memory helps even within a single extended task. How: This would need to be distinguished from the cross-session memory system the user has explicitly disabled. A mission-scoped `scratch/` directory that gets committed with findings (or discarded) would serve a different purpose — working memory, not persistent memory. Impact: medium. Confidence: speculative.
